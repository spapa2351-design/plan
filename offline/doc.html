Governance & Workflow

에셋 거버넌스: 편집 제한이 아닌 '안전한 관리'

"편집을 아예 못하게 하는 것은 불가능합니다." 핵심은 에셋의 생애주기(Status)를 정의하고, 학습 데이터가 쌓인 시점부터 **수정의 파급효과를 운영자에게 경고**하는 것입니다.

📝
초안 상태 (Draft)
수강생이 아직 없습니다. 에셋의 파일, 제목, 설정을 무제한으로 수정할 수 있습니다.
🚀
활성 상태 (Published)
수강 데이터가 쌓이기 시작했습니다. 수정 시 **'전체 업데이트'** 혹은 **'사본으로 분리'**를 선택해야 합니다.
📦
보관 상태 (Archived)
더 이상 사용하지 않지만, 과거의 수강 통계를 보존하기 위해 시스템 뒤편에 영구 저장됩니다.

Pragmatic Proposal

실무적 타협안 제안: 진도율 무결성을 지키는 3가지 아이디어

완벽한 버전 관리(Versioning)는 자칫 개발 복잡도를 극심하게 높일 수 있습니다. 실제 현업의 에셋 재활용 빈도가 그리 높지 않다는 점을 감안하여, 개발 리소스를 대폭 줄이면서도 운영 리스크를 방어할 수 있는 현실적인 아이디어를 기획 단에서 제안해 봅니다. 개발팀이 보시기에 어떤 방식이 가장 적합할까요?

💡
제안 1 (가장 추천)
'영상' 대신 '목차(Slot)' 기준 매핑
"A영상 v1, v2를 개별 추적하지 말고, '1강 목차'라는 빈 껍데기(Slot)를 통과했는지만 체크하면 어떨까요? 나중에 1강 영상을 아예 다른 것으로 교체해도, 이미 1강을 수강 완료한 기존 학습자의 진도율은 100%로 인정해 주는 방식입니다."
💡
제안 2
에셋을 '참조' 대신 '템플릿(복제)'으로
"에셋 재활용률이 낮다면 무리해서 참조 모델을 고집할 필요 없이, 라이브러리의 원본을 코스로 가져올 때 해당 코스만을 위한 '독립된 사본'이 생성되게 하면 어떨까요? 기존의 단방향 DB 구조와 유사해 개발 공수가 크게 줄어들 것 같습니다."
💡
제안 3
수강생 발생 시 '수정/삭제 원천 차단'
"로직으로 풀지 않고 운영 정책으로 막는 가장 심플한 방법입니다. 수강 이력이 1건이라도 생기면 해당 에셋을 시스템적으로 수정 불가(Lock) 처리합니다. 꼭 수정해야 한다면, 기존 에셋을 숨김 처리하고 새 챕터를 추가하도록 가이드합니다."

전략적 실행 로드맵 (Execution Plan)

안정적인 아키텍처 전환을 위해 3단계 실행 로드맵을 제시합니다.

Phase 1

🏢 오프라인 코스 선(先) 개발

기존 온라인 구조를 건드리지 않고, '현대적 UI/UX'의 뼈대와 출결 모듈을 안전하게 확립합니다.

Game Changer
Phase 2

💻 온라인 코스 (가벼운 조립)

무거운 저작 도구를 탈피하고, 가벼운 '에셋 참조형 플레이리스트' 세팅을 구현합니다.

Phase 3

🔄 블렌디드 코스 완성

독립 모듈로 완성된 오프라인 출석과 온라인 콘텐츠를 하나의 커리큘럼 UI에서 융합합니다.
