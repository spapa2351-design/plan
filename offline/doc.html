Core Architectural Shift

페이지에 갇힌 콘텐츠를
독립된 자산(Asset)으로 해방하다

기존의 '이북(e-book)' 형태의 무거운 페이지 저작 도구에서 벗어납니다. 콘텐츠는 레고 블록처럼 중앙 보관함에 단 1개만 존재하며, 수백 개의 코스에서 이를 동시 참조(Link)하여 조립하는 가장 현대적인 플레이리스트(Playlist) 아키텍처를 소개합니다.

Terminology Shift

명칭의 재정의: 페이지에서 아이템으로

새로운 아키텍처를 명확히 이해하기 위해, 시스템에서 부르는 이름부터 '학습 아이템(Asset)'으로 변경합니다.

인터랙티브 시뮬레이터: 복제 vs 참조

아래 시뮬레이터는 원본 영상에 수정 사항이 생겼을 때, 과거의 방식(Legacy)과 새로운 에셋 참조 방식(Next-Gen)이 운영자의 업무량에 어떠한 차이를 가져오는지 보여줍니다.

AS-IS (복제 모델)

물리적 페이지 종속 구조

영상이 각 코스의 '페이지' 내부에 물리적으로 복사되어 갇혀 있습니다. 수정 시 모든 코스를 찾아다녀야 합니다.

코스 A의 페이지
구버전 영상 (v1)
코스 B의 페이지
구버전 영상 (v1)
코스 C의 페이지
구버전 영상 (v1)
⚠️ "코스가 100개면 운영자는 이 짓을 100번 반복해야 합니다."
TO-BE (참조 모델)

에셋(Asset) 논리적 링크 구조

단 하나의 마스터 에셋만 존재하며, 각 코스는 이 에셋의 ID만 참조합니다. 1회 수정으로 완벽한 동기화가 이루어집니다.

Risk Mitigation

공유 에셋의 사이드 이펙트(Side-effect) 원천 차단

"참조 모델의 가장 큰 위험은 의도치 않은 다른 코스에 영향을 주는 것입니다. 예를 들어 코스 A만을 위해 영상을 수정했는데, 이를 공유 중인 코스 B와 C까지 변경되어 대형 운영 사고가 발생할 수 있습니다. 이를 방어하기 위해 시스템은 공유 중인 에셋 수정 시 지능형 버전 관리(분기) 팝업을 강제로 띄웁니다."

- 전사 공통 법정의무교육

데이터 임팩트: 정량적 효과 분석

아키텍처가 전환될 때, 플랫폼 운영과 인프라 관점(유지보수 시간 및 스토리지 공간)에서 발생하게 되는 압도적인 비용 절감 효과를 차트로 증명합니다.

콘텐츠 유지보수 소요 시간

동일 영상이 100개 코스에 포함되었을 때 업데이트 기준

시스템 스토리지 누적 데이터량

코스 생성 및 콘텐츠 재사용에 따른 물리적 용량 증가 추이

전략적 실행 로드맵 (Execution Plan)

안정적인 아키텍처 전환을 위해 3단계 타당성 검토 기반의 실행 로드맵입니다. 기존 시스템의 얽힌 레거시(Legacy)를 안전하게 회피하면서도 고객의 시급한 요구사항을 충족시키는 개발 순서를 제시합니다.

Phase 1

🏢 오프라인 코스 선(先) 개발

기존 온라인 DB를 건드리지 않고, '쉽고 편한 현대적 UI/UX'의 뼈대와 출결 내재화 모듈을 안전하게 테스트하고 확립합니다.

GAME CHANGER
Phase 2

💻 온라인 코스 (가벼운 조립)

무거운 저작 도구를 완전히 탈피하고, 가벼운 '에셋 참조형 플레이리스트' 세팅을 구현하여 폭발적인 반응을 이끌어냅니다.

Phase 3

🔄 블렌디드 코스 완성

독립적인 모듈로 단단하게 완성된 오프라인 출석과 온라인 콘텐츠를 하나의 커리큘럼 UI 안에서 섞어주는(Mix) 융합 단계입니다.
