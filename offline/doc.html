Core Architectural Shift

페이지에 갇힌 콘텐츠를
독립된 자산(Asset)으로 해방하다

기존의 '이북(e-book)' 형태의 무거운 페이지 저작 도구에서 벗어납니다. 콘텐츠는 레고 블록처럼 중앙 보관함에 단 1개만 존재하며, 수백 개의 코스에서 이를 동시 참조(Link)하여 조립하는 가장 현대적인 플레이리스트(Playlist) 아키텍처를 소개합니다.

인터랙티브 시뮬레이터: 복제 vs 참조

아래 시뮬레이터는 시스템 내부의 작동 방식을 시각화합니다. 원본 영상에 수정 사항이 생겼을 때, 과거의 방식(Legacy)과 새로운 에셋 참조 방식(Next-Gen)이 운영자의 업무량에 어떠한 차이를 가져오는지 직접 버튼을 눌러 비교해 보십시오.

AS-IS (복제 모델)

물리적 페이지 종속 구조

영상이 각 코스의 '페이지' 내부에 물리적으로 복사되어 갇혀 있습니다. 수정 시 모든 코스를 찾아다녀야 합니다.

구버전 영상 (v1)

구버전 영상 (v1)
코스 C의 페이지
구버전 영상 (v1)
⚠️ "코스가 100개면 운영자는 이 짓을 100번 반복해야 합니다."
TO-BE (참조 모델)

에셋(Asset) 논리적 링크 구조

단 하나의 마스터 에셋만 존재하며, 각 코스는 이 에셋의 ID만 참조합니다. 1회 수정으로 완벽한 동기화가 이루어집니다.

데이터 임팩트: 정량적 효과 분석

'페이지' 기반에서 '에셋 참조형 플레이리스트'로 아키텍처가 전환될 때, 플랫폼 운영과 인프라 관점(유지보수 시간 및 스토리지 공간)에서 발생하게 되는 압도적인 비용 절감 효과를 정량적인 차트로 증명합니다.

콘텐츠 유지보수 소요 시간

동일 영상이 100개 코스에 포함되었을 때 업데이트 기준

Insight: 기존에는 100개의 에디터를 일일이 열어 영상을 교체해야 하므로 기하급수적인 노가다 시간(약 500분)이 소요되었으나, 참조 구조에서는 중앙 에셋 1개만 수정(1분 이내)하면 수백 개의 코스에 즉시 동기화되어 공수가 99% 절감됩니다.

시스템 스토리지 누적 데이터량

코스 생성 및 콘텐츠 재사용에 따른 물리적 용량 증가 추이

Insight: 과거에는 코스를 복사할 때마다 내부에 임베드된 물리적 데이터(페이지 데이터)가 중복 생성되어 스토리지 낭비가 심했으나, TO-BE 모델은 메타데이터(Link ID)만 추가되므로 코스가 늘어나도 스토리지 부하가 거의 발생하지 않습니다.

전략적 실행 로드맵 (Execution Plan)

안정적인 아키텍처 전환을 위해 Canvas 문서에 정의된 3단계 타당성 검토 기반의 실행 로드맵입니다. 기존 시스템의 얽힌 레거시(Legacy)를 안전하게 회피하면서도 고객의 시급한 요구사항을 충족시키는 가장 현명한 개발 순서를 제시합니다.

Phase 1

🏢 오프라인 코스 선(先) 개발

복잡한 기존 온라인 DB를 건드리지 않고, 완전히 새로운 백지(빈 도화지) 상태에서 '쉽고 편한 현대적 UI/UX'의 뼈대와 출결 내재화 모듈을 안전하게 테스트하고 확립합니다.

Key Target: 리스크 최소화 및 물리적 일정 관리 UI 최적화
GAME CHANGER
Phase 2

💻 온라인 코스 (가벼운 조립)

무거운 페이지 저작 도구를 완전히 탈피하고, 파일(영상) 업로드 중심의 가벼운 '에셋 참조형 플레이리스트' 세팅을 구현하여 시장의 폭발적인 반응을 이끌어냅니다.

Key Target: 매출 이탈 방지 및 조립(Curation) 경험 극대화
Phase 3

🔄 블렌디드 코스 완성

독립적인 모듈로 단단하게 완성된 Phase 1(오프라인 출석)과 Phase 2(온라인 콘텐츠)를 하나의 커리큘럼 UI 안에서 섞어주는(Mix) 최종 융합 단계입니다.

Key Target: 온/오프 통합 완벽한 블렌디드 러닝 통제
