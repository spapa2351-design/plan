## 1. 블렌디드 코스 운영 케이스

### A사 케이스 — 순서형 블렌디드

```
[슬롯1] 온라인 영상 (오프라인 수업 전 선행 필수)
    ↓ 미시청 시 수료 불리 (soft 조건)
[슬롯2] 오프라인 출석
    ↓ 수업 당일 현장에서
[슬롯3] 온라인 시험 (현장 응시 후 제출)
    ↓ 제출 완료 후
[슬롯4] 온라인 설문
    ↓
  수료
```

- 수료 조건: 오프라인 출석 + 시험 제출 + 설문 완료
- 특이사항: 시험이 오프라인 현장에서 온라인으로 응시됨

---

### B사 케이스 — 게이트형 블렌디드 + 인사 연동

```
[슬롯1~10] 온라인 영상 10개
    ↓ 5개 이상 100% 시청 필수 ← 하드 게이트(Gate)
[슬롯11] 오프라인 출석 (게이트 미충족 시 참여 불가)
    ↓ 수업 다음날부터 시험 오픈
[슬롯12] 온라인 시험
    ↓ 80점 이상 ← 점수 커트라인
  수료 / 미수료
```

- 수료 조건: 영상 5개 이상 시청 + 오프라인 출석 + 시험 80점 이상
- 미수료 → 인사고과 반영
- 특이사항: 슬롯 간 게이트 — 앞 슬롯을 통과해야 다음 슬롯이 열림

---

## 2. 슬롯(Slot) 타입 정의

슬롯은 **관리자/수강생 눈에 보이지 않는 데이터 구조**. UI에서는 "커리큘럼 항목"으로 표현됨.

| 슬롯 타입 | 완료 조건 예시 | 비고 |
|---|---|---|
| 온라인 영상 | 시청률 100% | 부분 시청률 설정 가능 |
| 오프라인 출석 | QR 스캔 / 관리자 수동 처리 | Phase 1 핵심 |
| 온라인 시험 | 제출 완료 (A사) / 80점 이상 (B사) | 점수 기준 설정 가능 |
| 온라인 설문 | 제출 완료 | |

---

## 3. 완료 이벤트 추상화 — Phase 1에서 미리 잡을 것

블렌디드 Phase 3를 위해 Phase 1 오프라인에서 완료 이벤트를 슬롯 중심으로 기록해야 함.

```
❌ 지양: attendance { session_id, user_id, status }
   → 블렌디드 시 시험/영상 완료와 합산 불가

✅ 권장: completion_event { slot_id, user_id, type: 'attendance', created_at }
   → Phase 3에서 type: 'video_end' | 'exam_submit' 추가만 하면 됨
```

---

## 4. 게이트(Gate) 개념

B사 케이스에서 도출. 앞 슬롯 조건을 충족해야 다음 슬롯이 활성화되는 구조.

- Phase 1 오프라인에서는 필요 없음
- Phase 3 블렌디드에서 "영상 N개 이상 시청 → 오프라인 참여 가능" 구현 시 활용

---

## 5. 수료 조건 설계 방향

복합 조건을 지원하는 구조로 설계해야 함.

```
수료 조건 빌더 (미래 UI):
  [ ] 슬롯: 오프라인 출석 — 필수
  [ ] 슬롯: 온라인 시험 — 80점 이상
  [ ] 슬롯: 온라인 설문 — 제출 완료
```

Phase 1에서는 오프라인 출석 슬롯만 있지만, 구조는 확장 가능하게 잡아야 함.  
(현재 admin1.html에 수료 조건 설정 UI 없음 → 추가 필요)

---

## 6. "오프라인 먼저 설계" 전략 검토

**결론: 맞는 선택. 단, 블렌디드의 오프라인 모듈로 설계해야 함.**

- 오프라인이 블렌디드의 더 어려운 절반 (사람/공간/시간 동시 조건)
- 오프라인에서 잡은 패턴 (코스 관리 UI, 수강생 등록, 상태 관리)은 온라인/블렌디드에 재사용 가능
- 핵심: 출석 완료를 "슬롯 Pass 이벤트"로 기록하는 것이 Phase 3 연결의 열쇠

**Phase 1에서 당장 반영할 것:**
1. 완료 이벤트를 slot_id 기반으로 기록
2. 수료 조건 설정 UI 구조를 복합 조건 확장 가능하게 설계
3. 코스에 학점(Credit) 필드 추가 여부 검토 (C사 케이스 참고)

## 7. 용어 충돌 메모 (C사 케이스에서 발견, 미래 논의 사항)

| 용어 | 현재 admin1 의미 | 충돌 가능성 |
|---|---|---|
| 패키지 | 한 코스 내의 반/기수 (A반, B반, 1기) | C사에서 "패키지 코스" = 여러 코스를 묶은 번들 |

→ C사 구조 설계 전에 "패키지" 명칭 재정의 필요 (기수/반으로 변경 검토)
